# Goal 3.1 - Button feel rework

## Goal

Make physical and remote button behavior feel natural for museum visitors.
A button press should be easy to understand:

1. First press starts the room and turns the sound on quickly.
2. Immediate repeat pressing does not break anything and does not create odd toggles.
3. After a short minimum-on time, pressing the same button turns the sound off.
4. LED remains simple: on means the sound is playing, off means it is not playing.
5. Physical DIN buttons and web remote press use the exact same engine path.

This file replaces the earlier AI-generated note. It has been corrected against
the current codebase, which uses `config.json`, `src/audio_loop/core/looper_engine.py`,
`src/audio_loop/input/modbus_panel.py`, and a 16-sound room target.

## Review verdict

The original idea is good: the engine should own the context-aware button state,
not the Modbus polling layer. A per-instrument state machine is the right model.

Corrections made before implementation:

- The project uses `config.json`, not `config.yaml`.
- The room target is 16 sounds, not the old 18-layer prototype.
- `double_press_protection_ms` was per physical channel, not global. It is now
  removed because the engine state machine owns contextual repeat behavior.
- A 20 ms Modbus poll interval is not the production default. Current default
  is 50 ms until both boxes are measured together.
- The obsolete top-level `debouncing` config block was removed.
- Missing audio is ignored before starting the system or resetting the global
  timer. Pressing a button with no WAV file does not start an empty session.
- Compatibility-only button state was not kept. The state machine is now the
  engine source of truth.

## Target behavior

### Press behavior

| Current state | Press result | Global timer | Stats | LED |
| --- | --- | --- | --- | --- |
| `OFF_READY` and audio exists | activate layer | reset | record activation | on |
| `OFF_READY` but audio missing | ignore, log warning | no reset | no record | off |
| `ON_LOCKED` | ignore quietly | no reset | no record | on |
| `ON_READY` | deactivate layer | reset | no new activation | off |
| `OFF_COOLDOWN` | ignore quietly | no reset | no record | off |

### State meanings

```text
OFF_READY     - sound is off and can be started
ON_LOCKED     - sound is on, but cannot be turned off yet
ON_READY      - sound is on and can be turned off
OFF_COOLDOWN  - sound just turned off; very short re-arm delay
```

### State transitions

```text
OFF_READY --press with available audio--> ON_LOCKED
ON_LOCKED --min_on_seconds elapsed------> ON_READY
ON_READY --press------------------------> OFF_COOLDOWN
OFF_COOLDOWN --rearm_seconds elapsed----> OFF_READY

Any active state --instrument timeout---> OFF_COOLDOWN
Any state --global timeout/shutdown-----> OFF_READY
```

## Config target

Current `config.json` uses:

```json
{
  "inputs": {
    "provider": "modbus_panel",
    "min_on_seconds": 1.5,
    "rearm_seconds": 0.2
  },
  "modbus_panel": {
    "poll_interval_ms": 50,
    "debounce_time_ms": 40,
    "min_press_duration_ms": 10
  },
  "performance": {
    "max_concurrent_sounds": 16
  }
}
```

Removed from active config:

- `inputs.button_cooldown_seconds`
- `modbus_panel.double_press_protection_ms`
- top-level `debouncing` block

Notes:

- `poll_interval_ms = 50` is the production starting point.
- `poll_interval_ms = 20` and `debounce_time_ms = 30` are bench-test values only
  after measuring both Modbus boxes together.
- `min_on_seconds = 1.5` is intentionally short enough that the visitor does not
  feel trapped.
- `rearm_seconds = 0.2` avoids accidental double-taps while staying responsive.

## Implementation phases

### Phase 1 - Normalize max sound count

Status: `[implemented, verified] 2026-06-29 15:44:29 +02:00`

Implemented:

- Added configured `max_instruments` to `AudioManager`, `LooperEngine`, and
  `StatsCollector`.

  configured range.

Acceptance:

- `AudioManager` loads only `1.wav` through `16.wav` by default.
- `LooperEngine` status returns only instruments 1-16 by default.
- `POST /api/layers/17/press` is invalid with the current config.

### Phase 2 - Remove old timing guards

Status: `[implemented, verified] 2026-06-29 15:44:29 +02:00`

Implemented:

- Removed `button_cooldown_seconds` and `last_press_times` from `LooperEngine`.
- Removed `double_press_protection_ms` and `last_trigger_at` from
  `ModbusButtonHandler`.
- Kept DI debounce and minimum press duration in `ModbusButtonHandler`.
- Removed obsolete config keys from `config.json`.

Acceptance:

- Holding a physical button still triggers once on rising edge.
- Two different buttons pressed quickly can both reach `handle_button_press`.
- The engine, not Modbus polling, decides whether the press is meaningful.

### Phase 3 - Add per-instrument state machine

Status: `[implemented, verified] 2026-06-29 15:44:29 +02:00`

Implemented:

- Added `InstrumentState` enum in `looper_engine.py`.
- Added one state per instrument.
- Active layer status is derived from `ON_LOCKED` and `ON_READY`.
- Added activation/deactivation timestamps for timed transitions.
- `_logic_loop` and `handle_button_press` both advance timed transitions.

Acceptance:

- First press of available sound starts playback and enters `ON_LOCKED`.
- Press during `ON_LOCKED` is ignored and does not reset global timeout.
- After `min_on_seconds`, the same sound becomes `ON_READY`.
- Press during `ON_READY` fades the sound out and turns the LED off.
- Press during `OFF_COOLDOWN` is ignored.
- After `rearm_seconds`, the sound returns to `OFF_READY`.

### Phase 4 - Fix missing-audio behavior

Status: `[implemented, verified] 2026-06-29 15:44:29 +02:00`

Implemented:

- `handle_button_press` checks `audio_manager.get_available_instruments()`
  before activating the system from `OFF_READY`.
- Missing audio does not start playback, reset global timeout, increment stats,
  or turn on LED.
- Missing audio logs a warning and returns cleanly.

Acceptance:

- Pressing DI5 when `5.wav` is missing leaves the system idle if no other sound
  is active.
- Pressing a missing sound while the system is already active does not extend
  the session timer.

### Phase 5 - Keep LED and web behavior simple

Status: `[implemented, verified] 2026-06-29 15:44:29 +02:00`

Implemented:

- LED on for `ON_LOCKED` and `ON_READY`.
- LED off for `OFF_READY` and `OFF_COOLDOWN`.
- Web remote press continues to call `LooperEngine.handle_button_press(...)`.
- `/api/status` now includes `instrument_states` for developer/debug use; the
  operator dashboard remains simple.

Acceptance:

- Physical press and web press produce the same state transitions.
- Dashboard active state follows the engine state within the existing polling
  interval.
- LED state follows the actual active audio state.

### Phase 6 - Tests and bench verification

Status: `[implemented, smoke verified; Box 1 bench pending] 2026-06-29 15:44:29 +02:00`

Implemented tests:
